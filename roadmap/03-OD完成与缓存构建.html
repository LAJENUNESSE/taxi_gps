03 OD 完成与缓存构建

完成订单表、车辆缓存、分钟缓存和 OD 缓存，为后续地图展示与统计分析提供低延迟查询支撑。

返回总入口上一阶段下一阶段

完成内容

  • 基于 status 变化（0→1 上车，1→0 下车），用 shift(-1) 拼接 OD 对，生成 486,122 个出行订单。
  • 执行 src/07_缓存构建.py，对 38.9M 行 clean.csv 分块流式读取，逐车辆写入轨迹缓存。
  • 构建车辆元数据缓存 vehicles.json：记录每辆车的点数、字节偏移，支持 O(1) 抽取任意车辆轨迹。
  • 构建全量轨迹缓存 vehicle_data.json（约 2.6 GB），按 "vid": [[lon, lat, status, speed, hour], ...] 流式输出，无需加载整表到内存。
  • 构建车辆 ID 列表 vehicle_list.json，供地图页面下拉选择与轨迹查询接口调用。
  • 聚合统计缓存 stats_summary.json：按小时统计载客车数、平均速度，供热力图与统计图复用，避免重复扫描大文件。

输出文件

  • data/orders.csv — OD 出行表，486,122 行。
  • data/cache/vehicles.json — 车辆元数据（点数 + 偏移 + 来源文件）。
  • data/cache/vehicle_data.json — 全量轨迹缓存（~2.6 GB，流式可读）。
  • data/cache/vehicle_list.json — 车辆 ID 列表。
  • data/cache/stats_summary.json — 小时级载客车数与平均速度聚合。

运行方式

  • python src/02_OD提取.py — 生成 data/orders.csv（约 2 分钟）。
  • python src/07_缓存构建.py — 流式读 clean.csv 生成 4 个缓存文件，单次运行约 20 分钟。
  • 依赖 data/clean.csv 必须先由 01_数据清洗.py 产出。

难点

  • 全量轨迹缓存约 2.6 GB，不能一次性 json.load 进内存，必须按车辆键正则流式解析（见 10_路网分析.py 中的 _VEHICLE_KEY_RE）。
  • 分块读取同一辆车数据可能跨 chunk 边界，需缓存上一 chunk 末尾车辆继续拼接，避免轨迹断裂。
  • 元数据偏移必须与写入字节精确对应，否则后续轨迹查询会读到错位的 JSON 片段。
  • 小时级载客车统计与 04_可视化.py 的 occupied_taxis.csv 口径需一致，否则统计图会对不上。

注意事项

  • 缓存文件体积大，不要放入代码目录，统一存放 data/cache/。
  • vehicle_data.json 一旦重建，vehicles.json 中的偏移量必须同步重写，二者必须配套使用。
  • 如重跑清洗流程，所有缓存需一并删除重建，避免新旧数据不一致。
  • 缓存构建脚本应可重入：若中途崩溃，应能从车辆 ID 边界续跑，不必从头扫描 2.3 GB 清洗结果。
