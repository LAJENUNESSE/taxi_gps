**二、成员报告**

**1、成员A（组长）**

（1）本日进展

- 完成 `pipeline/cleaning.py` 数据清洗完整流水线的编码与调试，主要步骤如下：
  - **读取与排序**：使用 pd.read_csv 指定列名和 dtype（id→int32, long/lati→float32, status→int8, speed→int16），按 (id, time) 升序排序并重置索引。排序后进行单调性校验，检出 Time 非单调递减数。
  - **类型转换**：time 列从 HH:MM:SS 字符串转为 datetime 类型。
  - **坐标过滤**：剔除深圳范围（经度 113.5–114.8，纬度 22.3–22.9）外的记录。
  - **速度过滤**：剔除速度 < 0 或 > 120 km/h 的记录。
  - **跳点过滤**：按 id 分组，用 haversine 公式计算相邻点间距离，推导隐含速度，剔除 implied_speed > 150 km/h 的跳点。
  - **重复值去重**：对 (id, time) 重复记录分组，按 stat_cnt（2/3）和 stat_sum（各行的 status 之和）分 6 种情况分别处理。例如 cnt=2 且 sum=1 时保留 status=0 的行；cnt=3 且 sum=2 时保留 status=1 的行。
  - **异常状态检测**：通过 shift() 构造前后状态、前后车辆 ID、前后时间，筛选 60 秒内 status 跳变又跳回的短时间异常。
  - **车辆级过滤**：剔除速度始终为 0 的车辆和全天状态从未变化的车辆。
- 实际产出：clean_stage1.csv（46,927,855 行）→ clean_stage2.csv（46,529,551 行）→ clean.csv（38,694,096 行）。
- 所有输入输出路径、深圳边界、速度阈值等常量统一放在 config.py 中，代码不含硬编码绝对路径。

（2）问题汇报

- 重复值去重规则最初仅考虑了 cnt=2 和 cnt=3 的部分分支，后续补充了 cnt=3 且 sum=1（三条重复中仅一条 status=1）等边缘情况，避免漏删。
- 跳点过滤阈值从 200 km/h 下调至 150 km/h，原因是深圳城市道路条件下 200 km/h 无法过滤部分明显跳点。

**2、成员B**

（1）本日进展

- 完成 `pipeline/od_extraction.py` OD提取流水线的编码与运行验证，实际输出如下：
  - **status_chg 计算**：读取 clean.csv，将 time 转为 datetime。通过 shift(1) 构造 status_up 和 id_up，计算 status_chg = status - status_up 和 id_chg = id - id_up。
  - **上下车点识别**：status_chg == 1 为上车点（0→1），status_chg == -1 为下车点（1→0）。检出上车点 492,895 个，下车点 492,895 个。加入 id_chg == 0 同车过滤后保留 980,223 个。
  - **OD对拼接**：对筛选结果执行 shift(-1) 取出下一行的经纬度和时间作为下车信息。以 status_chg == 1 且 id == id.shift(-1) 为配对条件，成功配对 484,208 对。
  - **无效订单过滤**：删除 OD_TIME_s <= 0 的订单 0 条，删除 OD_Dis_km <= 0 的订单 1,810 条。
  - **最终输出**：orders.csv 共 482,398 个出行对，包含车辆ID、开始/结束时间、开始/结束经纬度、O_SPEED、D_SPEED、OD_TIME_s、OD_Dis_km 等 15 个字段。
- 所有输出路径集中配置，代码不含绝对路径。

（2）问题汇报

- 上车点和下车点数完全相等（各 492,895），说明数据清洗阶段已完整保留了每段载客行程的起止点，无首尾截断问题。
- 配对成功率：484,208 / 492,895 ≈ 98.24%，少数上车点无法配对的原因为数据末尾无对应的下车点记录（处于载客状态的车辆数据截断）。
- 距离为 0 的无效订单 1,810 条被过滤，原因是同一地点立即下车或 GPS 精度导致的经纬度未变化。
